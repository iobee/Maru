# 应用动态与后台日志分层设计

日期：2026-07-12  
状态：已实现，空状态与后台日志已完成真实界面验收  
范围：新增独立“应用动态”模块，并重新定位和优化“后台日志”模块

## 1. 产品目标

把过去混在一个日志页面里的两种需求彻底拆开：

- `应用动态` 面向普通用户，用于按 App 理解 Maru 为什么介入、操作了哪个窗口、执行结果是什么。
- `后台日志` 面向开发者诊断，保留完整技术细节，让用户能方便地复制或导出后发送给开发者。

应用动态不是后台日志的文本过滤器。它使用独立的结构化事件模型，避免依赖易变化的日志文案。

## 2. 导航与页面定位

侧栏顺序：

1. 常规
2. 快捷键
3. 应用规则
4. 应用动态
5. 后台日志
6. 关于

两个页面都属于高密度工具页，使用稳定实体表面、轻描边和内部滚动，不使用大面积玻璃或彩色卡片阵列。

## 3. 应用动态

### 3.1 信息结构

- 左侧 App 选择器：App 图标、名称、动态数量、最近活动时间、名称与 Bundle ID 搜索。
- 右侧时间线：时间、事件类型、用户可理解的标题与原因、窗口标题、触发来源。
- 默认选择最近产生动态的 App。

### 3.2 记录范围

- App 启动
- App 进入前台
- 即使辅助功能权限未授予或窗口自动管理未运行，也继续记录 App 启动、进入前台，并明确说明没有检查窗口的原因
- 新窗口或焦点窗口变化
- 等待窗口信息就绪
- 准备执行居中、呼吸窗口或移动显示器
- 因辅助窗口、多窗口歧义、忽略规则、冷却期等原因跳过
- 操作成功或失败
- 手动操作的请求与结果

### 3.3 数据与隐私

- 数据只保存在本机 `app_activity.json`。
- 最多保留最近 1200 条，超出后自动删除最旧记录。
- 用户可以从页面右上角清除全部应用动态。
- 清除应用动态不会删除后台日志；清空内存日志也不会删除应用动态。

## 4. 后台日志

### 4.1 页面职责

后台日志不再承担用户理解窗口行为的任务，只提供开发者排查所需的原始技术信息。

### 4.2 体验要求

- 用逐行结构替代整块横向滚动文本。
- 每行明确显示时间、级别、消息、源码文件与行号。
- 支持按最低日志级别和文本搜索筛选当前视图。
- `复制日志` 与 `导出` 始终包含当前来源的完整日志，避免用户因筛选遗漏关键上下文。
- 历史文件载入到独立查看状态，不覆盖或污染实时内存日志。
- 清空实时内存日志放入次级菜单，并明确不会删除磁盘历史文件。

## 5. 验收标准

- 侧栏同时存在独立的“应用动态”和“后台日志”。
- 应用动态能按 App 搜索和切换，并只展示所选 App 的结构化事件。
- 时间线能说明触发来源、目标窗口、执行或跳过原因和最终结果。
- 动态跨应用重启持久化，且数量上限生效。
- 后台日志行可读、可搜索、可按级别过滤。
- 复制和导出提供完整日志；历史日志不会覆盖实时日志。
- 页面符合 `docs/PRODUCT-UI-DESIGN-STANDARD.md`。
- 单元测试、全量测试、release build 和真实 `.app` 界面验收通过。
