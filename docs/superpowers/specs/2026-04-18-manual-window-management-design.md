# HiWindowGuy 手动窗口管理能力 Spec

日期：2026-04-18  
状态：已完成设计确认，待实现  
范围：新增手动窗口管理动作、全局快捷键、菜单入口和首页快捷键设置区

## 1. 目标

为 HiWindowGuy 新增两项“手动窗口管理”能力，并确保它们既能通过全局快捷键触发，也能通过菜单栏入口和应用菜单触发。

新增动作：

- 窗口居中
- 几乎最大化

本次设计的核心目标：

- 在不影响现有自动规则链路的前提下，新增一条独立的手动触发链路
- 手动触发的目标窗口选择规则，必须符合经典窗口管理软件的直觉
- 全局快捷键可在 HiWindowGuy 不在前台时使用
- 快捷键可配置、可清除、可恢复默认，并且不能互相冲突
- 新增设置 UI 必须严格服从现有产品 UI 规范，不能破坏首页现有风格

## 2. 用户已确认的设计决策

- 实现方向：独立的手动窗口管理链路，不和自动规则入口混用
- 手动选窗逻辑：按前台 app / 焦点标准窗口，不按鼠标悬停窗口
- 配置入口位置：放在现有“常规”首页
- 失败反馈：无法找到可操作窗口时，弹出轻量提示
- UI 约束：必须严格保持与现有 UI 规范一致

## 3. 范围边界

### 3.1 本次包含

- 新增手动动作：窗口居中、几乎最大化
- 新增全局快捷键注册与配置能力
- 新增菜单栏下拉菜单入口
- 新增顶部应用菜单“窗口管理”入口
- 新增首页“常规”中的手动快捷键设置区
- 新增手动链路的失败提示与日志记录

### 3.2 本次不包含

- 现有自动规则链路的整体重构
- 自动规则的鼠标优先选窗逻辑重做
- 新增独立设置页或新增导航项
- 首页 Hero 控制卡或侧栏结构方向变更
- 新增通知系统、浮层系统或复杂状态中心

## 4. 问题诊断

HiWindowGuy 当前只有“自动触发”能力，缺少用户主动发起的窗口操作入口。现有 `WindowManager` 自动链路的选窗逻辑是为“应用切换后自动处理”设计的，核心是鼠标优先选窗，再 fallback 到焦点窗口和窗口列表。

这套逻辑不适合手动快捷键，原因有三点：

- 用户按下快捷键时，期望操作的是当前前台 app 的活动标准窗口，而不是鼠标悬停位置下的窗口
- 快捷键行为需要与主流窗口管理软件保持一致，目标判断必须更可预测
- 自动链路和手动链路的触发来源、目标窗口定义和错误反馈都不同，不应共享同一个入口分支

因此，本次设计要求把“最终执行窗口居中 / 几乎最大化”与“如何解析目标窗口”分层处理：底层窗口调整能力可复用，手动触发入口必须独立。

## 5. 设计原则

### 5.1 手动链路独立

- 手动触发与自动规则共享底层窗口调整实现
- 手动触发不复用自动规则入口
- 手动触发不读取鼠标位置，不使用鼠标优先选窗

### 5.2 入口统一，行为一致

- 全局快捷键
- 菜单栏下拉菜单
- 顶部应用菜单

以上三个入口必须调用同一套手动动作分发逻辑，确保行为完全一致。

### 5.3 设置内聚

- 快捷键配置集中归属到现有首页“常规”页
- 配置持久化归属 `AppConfig`
- 全局快捷键监听归属专门的管理器，不散落在视图层

### 5.4 UI 严格服从现有规范

- 必须遵循 `docs/PRODUCT-UI-DESIGN-STANDARD.md`
- 必须与 `docs/superpowers/specs/2026-04-15-homepage-control-panel-design.md` 的首页结构保持一致
- 不得因为新增设置区而破坏首页主次层级

## 6. 最终交互定义

### 6.1 动作定义

新增两个手动动作：

- `窗口居中`
- `几乎最大化`

动作语义：

- `窗口居中`：对目标窗口执行居中，不改变其当前大小
- `几乎最大化`：对目标窗口执行现有“几乎最大化”逻辑，继续使用当前配置中的缩放比例

### 6.2 默认快捷键

- 窗口居中：`Control + Command + C`
- 几乎最大化：`Control + Command + M`

### 6.3 触发入口

#### A. 全局快捷键

- HiWindowGuy 不在前台时也可触发
- 快捷键触发行为与菜单点击一致

#### B. 菜单栏下拉菜单

新增两个入口：

- `窗口居中`
- `几乎最大化`

#### C. 顶部应用菜单中的“窗口管理”菜单

新增两个入口：

- `窗口居中`
- `几乎最大化`

菜单入口与全局快捷键必须调用同一动作，不允许出现菜单和快捷键行为不一致的情况。

## 7. 手动链路的目标窗口解析规则

### 7.1 目标定义

手动动作的作用目标定义为：

`当前前台 app 的活动标准窗口`

这意味着：

- 以 `NSWorkspace.shared.frontmostApplication` 作为唯一 app 入口
- 不看鼠标悬停窗口
- 不跨应用查找窗口

### 7.2 解析顺序

手动链路按以下顺序解析目标窗口：

1. 获取当前前台 app
2. 读取该 app 的 `AXFocusedWindow`
3. 如果焦点窗口是标准窗口，则直接使用
4. 如果焦点窗口不存在或不可用，则 fallback 到该 app 的窗口列表
5. 从窗口列表中选取第一个可操作的标准窗口
6. 如果仍然找不到，则给出失败提示并终止操作

### 7.3 标准窗口判断

手动链路沿用现有项目中对标准窗口的基本判断思路，但只在“前台 app 范围内”生效：

- 接受 `AXStandardWindow`
- 接受少数没有明确 subrole 但位置和大小可读写的标准业务窗口
- 跳过最小化窗口
- 排除明显不可操作的非标准辅助窗口

### 7.4 明确禁止

手动链路明确禁止以下行为：

- 使用鼠标坐标查找窗口
- 用鼠标下元素向上追溯窗口
- 因鼠标停在别的 app 上而选中非前台 app 的窗口
- 直接复用当前自动规则入口函数作为手动入口

## 8. 错误处理与用户反馈

### 8.1 无法找到窗口

当手动触发时，如果当前前台 app 没有可操作的标准窗口，应弹出轻量 `NSAlert`。

建议文案：

- 标题：`无法找到可操作的窗口`
- 说明：`当前前台应用没有活动的标准窗口，或该窗口不支持窗口管理。`

### 8.2 权限问题

如果缺少辅助功能权限：

- 沿用现有权限检查与提示逻辑
- 不新增第二套权限交互

### 8.3 日志

手动动作应写入日志，至少记录：

- 触发来源（全局快捷键 / 菜单栏 / 应用菜单）
- 动作类型（居中 / 几乎最大化）
- 前台 app 名称与 bundle id
- 成功 / 失败结果
- 失败原因（无权限 / 无窗口 / 设置冲突后未注册等）

## 9. 配置模型设计

### 9.1 配置项

为两个固定动作新增两个快捷键配置项：

- `manualCenterShortcut`
- `manualAlmostMaximizeShortcut`

每个配置项保存：

- 主键
- 修饰键集合

### 9.2 默认值

- `manualCenterShortcut` 默认值：`Control + Command + C`
- `manualAlmostMaximizeShortcut` 默认值：`Control + Command + M`

### 9.3 支持操作

每个快捷键配置必须支持：

- 查看当前绑定
- 修改
- 清除
- 恢复默认值

### 9.4 冲突约束

禁止将两个动作绑定为同一个组合键。

冲突处理规则：

- 用户提交重复组合键时，不保存配置
- 不重新注册对应快捷键
- 在设置 UI 上给出明确错误提示

### 9.5 持久化归属

快捷键配置应归属到 `AppConfig` 的一般配置持久化范围中，与 `windowScaleFactor` 一起保存在 `general.json`。

## 10. 组件与职责划分

### 10.1 WindowManager

职责：

- 对外暴露两个手动动作入口
- 校验权限
- 请求手动目标窗口
- 调用现有 `centerWindow` / `almostMaximizeWindow`
- 负责失败提示与手动链路日志

新增公开入口建议：

- `performManualCenter(triggerSource:)`
- `performManualAlmostMaximize(triggerSource:)`

### 10.2 ManualWindowTargetResolver

职责：

- 只负责解析“当前前台 app 的活动标准窗口”
- 封装手动链路的选窗规则
- 与自动链路的鼠标优先选窗逻辑彻底分离

说明：

- 可以作为 `WindowManager` 内部私有实现拆分
- 也可以抽成独立类型
- 关键要求是职责明确，不能和自动链路混在同一入口函数中

### 10.3 HotkeyManager

职责：

- 注册全局快捷键
- 响应快捷键事件
- 配置变更后重新注册
- 处理清除后的注销
- 作为快捷键监听与动作分发之间的桥梁

### 10.4 AppConfig

职责：

- 持有快捷键配置
- 提供默认值、清除、恢复默认等能力
- 负责冲突校验
- 持久化到 `general.json`

### 10.5 HomeDashboardView

职责：

- 新增“手动窗口管理”设置卡
- 展示当前绑定
- 提供修改、清除、恢复默认入口
- 展示冲突和录制状态

注意：

- 视图层只处理配置编辑和展示
- 不直接实现全局快捷键注册

## 11. 首页 UI 设计要求

### 11.1 位置

新增的“手动窗口管理”设置区放在现有“常规”首页中，作为一张新的常规设置卡接入。

首页仍保持现有三层主次关系：

1. 页面头部
2. 主控制卡
3. 常规设置卡

新增设置卡属于次级控制区域，不能抢 Hero 主控制卡的视觉中心。

### 11.2 样式要求

该设置卡必须严格服从当前产品 UI 规范：

- 使用常规设置卡样式，不新增网页式设置面板
- 保持现有常规卡的深色实体表面
- 保持现有常规卡的圆角、内边距、轻描边和蓝色强调体系
- 不新增大面积玻璃
- 不新增彩色装饰块、渐变背景或悬浮式快捷键配置器

### 11.3 行级结构

每个动作建议为一行，结构为：

- 动作名称
- 当前快捷键显示
- 行内操作按钮：修改 / 清除 / 恢复默认

若进入录制态，应采用克制的行内编辑态，不引入夸张弹层。

### 11.4 错误与状态表达

- 冲突提示使用现有语义色体系
- 不新增第二套强调色
- 状态文案应清晰，但视觉权重低于卡片标题和当前绑定展示

## 12. 菜单设计要求

### 12.1 菜单栏下拉菜单

保持现有结构，只新增两项动作入口，不改菜单整体方向。

### 12.2 顶部应用菜单“窗口管理”

保持现有菜单结构，新增两项动作入口，与菜单栏入口一致。

### 12.3 一致性要求

- 菜单文案一致
- 入口顺序一致
- 行为一致
- 不在不同菜单中提供不同版本的动作逻辑

## 14. 验收标准

完成后应满足以下条件：

- 用户可以通过全局快捷键触发“窗口居中”和“几乎最大化”
- HiWindowGuy 不在前台时，全局快捷键仍可工作
- 菜单栏下拉菜单可直接触发两项动作
- 顶部“窗口管理”菜单可直接触发两项动作
- 手动动作的目标窗口来自前台 app 的活动标准窗口，而不是鼠标悬停窗口
- 快捷键可在首页查看、修改、清除、恢复默认
- 两个动作不能绑定为相同组合键
- 快捷键配置区视觉上与当前首页严格一致，不像后贴进去的新系统
- 现有自动规则链路继续保留并正常工作

## 15. 测试关注点

- 全局快捷键在 app 前台和后台两种情况下都能触发
- 修改快捷键后重新注册生效
- 清除快捷键后对应组合键失效
- 恢复默认后恢复原始绑定
- 冲突绑定被阻止，且 UI 提示正确
- 前台 app 有焦点标准窗口时，手动动作命中正确目标
- 前台 app 无焦点标准窗口时，fallback 到窗口列表成功
- 前台 app 无可操作标准窗口时，弹出失败提示
- 自动规则链路不受回归影响
